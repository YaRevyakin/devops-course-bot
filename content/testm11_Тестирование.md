### 1. Зачем разделять VPC на публичные и приватные подсети?
- [ ] A) Чтобы ускорить интернет  
- [ ] B) Чтобы снизить стоимость трафика  
- [ ] C) Чтобы изолировать критические сервисы (например, БД) от прямого доступа из интернета  
- [ ] D) Чтобы упростить развёртывание WordPress

<details>
<summary>Показать ответ</summary>

**C** — публичные подсети используются для компонентов, доступных из интернета (балансировщики, веб-серверы), а приватные — для внутренних сервисов (БД, кэш), которые не должны быть напрямую доступны.
</details>

### 2. Какой сервис лучше всего подходит для запуска редко используемой функции обработки загруженных файлов?
- [ ] A) Managed Kubernetes  
- [ ] B) Виртуальная машина с cron  
- [ ] C) Serverless Functions (Lambda / Cloud Functions)  
- [ ] D) Выделенный сервер

<details>
<summary>Показать ответ</summary>

**C** — serverless идеально подходит для event-driven задач: оплата только за время выполнения, автоматическое масштабирование от 0, нет необходимости управлять инфраструктурой.
</details>
---
### 3. Что такое private endpoint в контексте managed-базы данных?
- [ ] A) Публичный URL для доступа к БД  
- [ ] B] Приватный IP-адрес внутри VPC, позволяющий обращаться к БД без выхода в интернет  
- [ ] C) SSH-туннель к БД  
- [ ] D) Резервная копия БД

<details>
<summary>Показать ответ</summary>

**B** — private endpoint обеспечивает безопасное подключение к managed-сервису (БД, хранилище) **только из вашей VPC**, без публичного IP и без риска перехвата трафика.
</details>

### 4. Почему стоит использовать managed Kubernetes вместо self-hosted?
- [ ] A) Он дешевле  
- [ ] B) Он не требует worker nodes  
- [ ] C] Провайдер управляет control plane, обеспечивая отказоустойчивость и обновления без вашего участия  
- [ ] D) Он работает только с Docker

<details>
<summary>Показать ответ</summary>

**C** — в managed Kubernetes (EKS, GKE, Yandex Managed K8s) провайдер берёт на себя управление мастер-нодами, что снижает операционную нагрузку и повышает надёжность.
</details>

### 5. Какой принцип безопасности требует давать сервисам только минимально необходимые права в облаке?
- [ ] A) Defense in Depth  
- [ ] B) Zero Trust  
- [ ] C] Least Privilege  
- [ ] D] Shared Responsibility

<details>
<summary>Показать ответ</summary>

**C** — принцип наименьших привилегий (Least Privilege) минимизирует последствия компрометации: даже если злоумышленник получит доступ к сервису, его возможности будут ограничены.
</details>
